*A technical reference for infrastructure engineers entering AI operations*

## Note on Units of Measurement

This manual uses **GB (gigabytes)** and **MB (megabytes)** following the common convention in AI/ML documentation and hardware datasheets. When referring to "70 GB for a 70B model in INT8", we mean 70 billion bytes (70 × 10⁹).

Most documentation (NVIDIA, PyTorch, cloud providers) uses GB/MB ambiguously, sometimes referring to binary units (GiB = 1024³ bytes) and sometimes to decimal units (GB = 10⁹ bytes). We align with standard industry language, understanding that GB/MB represent approximate memory capacity.

70 decimal GB (70 × 10⁹ bytes) correspond to ~65.2 binary GiB. This ~7% difference is acceptable for capacity planning and architectural reasoning — it is smaller than the overhead introduced by frameworks (PyTorch/vLLM), memory fragmentation, and OS dynamic allocations.

# Chapter 0: The Birth of a Model

An LLM is not born intelligent. At first it is just an empty mathematical structure — a labyrinth of billions of random numbers. It becomes capable of generating text through a rigorous training process that transforms statistics into language. This chapter explores the lifecycle of a decoder-only Transformer — the architecture that dominates production today.

## The Base Structure: The Blueprint

Before anything else, the architecture must be defined: a neural network composed of dozens of identical stacked layers.

**Design choices**

Before training begins, three architecture choices directly impact runtime performance:

- **Hidden dimension:** the "width" of each layer. Determines the size of weights and the computational cost per token processed.
- **Number of layers:** the "depth" of the model. Determines latency — data must pass through every layer in sequence, without skipping any.
- **Context length:** the maximum sequence length the model can process. It is hardwired into the architecture during training and cannot be extended at deployment. Determines the maximum KV Cache size at runtime.

**Impact on VRAM:** These choices define the physical "weight" of the model. The wider the model, the more space it will occupy in GPU **VRAM** — both for weights and for the **KV Cache** during use.

## The Training Mechanism: The Error Signal

There is no training without a comparison. Training is a mathematical optimization loop based on answer verification:

1. **Forward Pass (Prediction):** The model receives a sequence and attempts to predict the next word.
2. **The Check (Loss Calculation):** The system compares the prediction with the correct answer.
3. **Backpropagation:** The error is sent "backwards" through the layers. The system calculates how much each weight contributed to the error.
4. **Optimizer:** The weights are slightly modified to reduce the error on the next attempt.

**In summary:** Training is a continuous dialogue between the model and a "judge" that holds the truth.

## Pre-training: Self-Supervised Learning

The model, initialized with random weights, is exposed to trillions of tokens of real text. For each sequence, the framework hides the last token and challenges the model to predict it. Weights are updated billions of times.

**Who does the check?** The training framework operates in **self-supervised** mode. It takes a real sentence, hides part of it, and challenges the model to reconstruct it. Since the software holds the original text, it acts as a mechanical judge.

## Supervised Fine-Tuning: The Teacher's Control

The pre-trained model is exposed to curated datasets of ideal question/answer pairs. It learns the conversation format and how to follow instructions.

**Who does the check?** Human experts who write "Ideal Q&A" datasets.

## Reinforcement Learning from Human Feedback: The Critic's Control

The model generates multiple response variants to the same question. Humans (or reward models) compare and vote on the best one. Weights are updated to favor preferred behaviors.

## Checkpoint: The Model is "Frozen"

At the end of all training phases, the final parameters are saved in a file called a **Checkpoint**.

- **The Watershed:** The dialogue ends here. The check disappears because it is no longer needed for learning. Backpropagation is turned off and weights become static.
- **Start of Inference:** The model is now a frozen object that only executes prediction computations. Our job as **Inference Engineers** is to take this object and optimize it for maximum speed.

### Training vs Inference — Structural Note

During training, every sequence is built on real text (ground truth). The model never sees its own outputs as context.

During inference, context accumulates including tokens generated by the model itself. Every produced token becomes input for the next — a regime called **autoregressive generation**.

This difference explains why a model can degrade on long sequences: it is operating in a domain (self-generated context) structurally different from what it was trained on.

## 0.1 The Frozen Model: Dimensions and Overhead

At the end of training, the model changes state: from modifiable to immutable.

In inference, all complexity disappears. Weights are fixed, loaded once into memory as read-only data. The size depends only on how many bytes you use to represent each parameter:

```
70B params × 2 bytes (FP16) = 140 GB + ~20 GB overhead = 160 GB (Standard)
70B params × 1 byte  (INT8) =  70 GB + ~10 GB overhead =  80 GB (Balanced)
70B params × 0.5 byte (INT4) = 35 GB +  ~5 GB overhead =  40 GB (Optimized)
```

VRAM does not only host model weights — subsequent chapters cover what else occupies it during execution.

---

## 0.2 The Model Container: Where Weights Live

The file format is the box in which the model is stored. The file extension is not just a label — it determines the loading strategy and the efficiency with which bytes travel from disk to GPU circuits.

### Format Selection: Loading Strategies

- **Raw Weights (.bin / .pt):** PyTorch's old standard based on Pickle. Requires Python to unpack. CPU must read data, reconstruct it in RAM, then send it to the GPU. Slow, requires much system RAM, and presents security risks.

- **Safe Weights (.safetensors):** The current enterprise performance and security standard. Via Memory Mapping (`mmap`), the inference engine can map weights directly from disk to GPU VRAM, bypassing intermediate CPU RAM steps. Practical advantage: near-instant startup and the ability to load giant models on machines with very little system RAM.

- **Compact Weights (.gguf):** The universal format for flexible inference (native to llama.cpp). Its main feature is the ability to split the model: if you have a 35 GB model and a 24 GB GPU, GGUF allows loading 20 GB on the GPU and "parking" the remaining 15 GB in system RAM.

#### Technical Box: Bypassing the CPU

In modern AI datacenters, the unifying principle behind every data transfer optimization is one: **eliminate the CPU from the data path.**

Technologies change, the problem is always the same:
- **Zero-copy / mmap** → direct storage to VRAM, CPU coordinates without touching data
- **GPUDirect RDMA** → network directly to VRAM, bypassing system RAM
- **NVLink** → GPU-to-GPU direct, without going through PCIe and CPU

The CPU remains essential as an orchestrator — it decides what to move and when — but no longer physically transports data.

### The Decision Fork: Algorithm vs Format

| **If the goal is...** | **Algorithm** | **Format** | **Engine (Layer 3)** |
|---|---|---|---|
| Scalable Production (Cloud) | AWQ / GPTQ | `.safetensors` | vLLM / TensorRT-LLM |
| Portability / Low VRAM | GGUF (K-Quants) | `.gguf` | llama.cpp / Ollama |
| Extreme Speed (Single GPU) | EXL2 | Folder/File | ExLlamaV2 |

### Case Study: Ollama vs vLLM

**Ollama — "On-Demand" Philosophy (Saving):** Default behavior is to load the model only when needed and unload it from VRAM after 5 minutes of inactivity. Every restart incurs a Cold Start cost.

**vLLM — "Always-On" Philosophy (Performance):** At startup, vLLM pre-loads not only **weights** but also pre-allocates all possible space for the **KV Cache**. The first response is instant because everything is already "warm" in GPU circuits.

## 0.3 Model Efficiency: Beyond Parameter Count

**The number of parameters represents potential "brain capacity"**, but effective intelligence depends on how that capacity is used.

Three reasons why a smaller-parameter model can beat a larger one:

**Data Quality:** A 7B model trained on 2 trillion tokens of high quality (books, clean code, logical reasoning) will almost always be more "intelligent" than a 70B model trained on junk data.

**Information Density (Chinchilla Scaling Laws):** DeepMind researchers discovered that many models were "under-trained." It is often more efficient to train a small model for much longer (exposing it to more data) rather than making a huge one and training it briefly.

**Architecture: Dense vs MoE (Mixture of Experts)**

- **Dense Models:** Every time you ask a question, all 70B parameters "activate."
- **MoE Models (e.g., Mixtral):** The model has many parameters (e.g., 47B), but for each word activates only a small portion (e.g., 12B). **Result:** You get the intelligence of a large model but the inference speed of a small one.

**Who selects which experts to activate?** Not a human, and not the inference software. The decision is **internal to the model** and was learned during training via a **Router** (Gating Network).

**Dense vs MoE Comparison:**

| **Characteristic** | **Dense (e.g., Llama 3 70B)** | **MoE (e.g., Mixtral 8x7B)** |
|---|---|---|
| Total Parameters | 70B | 47B |
| Active Parameters | **70B** (always all) | **12B** (only router-selected) |
| VRAM Cost | High (140 GB in FP16) | Medium-High (94 GB in FP16) |
| Speed (Token/s) | Slower | **Much faster** |

*Numbers rounded for clarity. Mixtral 8x7B has shared layers that bring it to ~46.7B effective parameters, with ~13B active per token.*

**Prefill vs Decode in MoE:**
- **Decode (generate tokens one by one):** This is where MoE expresses maximum advantage. Decode is memory-bound — reading 12B instead of 47B parameters is directly faster.
- **Prefill (process initial prompt):** The advantage is partial. Prefill is compute-bound — the GPU processes all prompt tokens in parallel and is already busy computing.

---

## 0.4 Architectures: Decoder, Encoder, Hybrid

**Decoder-only** (GPT, Llama, Mistral): generate text one word at a time, using a KV cache that grows linearly. These models dominate production because generative AI (chat, code, content creation) is the economically relevant workload.

**Encoder-only** (BERT): process input all at once, producing a fixed-size result (embeddings or classifications). Used for retrieval, classification, or embedding generation.

**Encoder-decoder** (T5, BART): combine both approaches. More complex to serve, primarily used for translation or summarization.

For the inference engineer, decoder-only represents over 90% of production work. The techniques discussed in this manual focus on this architecture.

## 0.5 The "Journey" Inside the Model

### Components

- **Parameters (The Cargo):** Numerical values (weights) that the model learned during training. Represent knowledge.
- **Layers (The Stations):** The model is divided into sequential floors. Data cannot skip floors.
- **Forward Pass (The Journey):** The complete execution. Starts from user input, traverses all layers, ends with the prediction of the next word. *This journey happens twice with different characteristics: once on the full prompt (prefill) and then once for each single token generated (decode).*

### Journey Optimization

#### 1. Hardware-Efficient Strategies (Optimize Movement)

- **Reduce weight (Quantization):** Converting parameters from FP16 to INT4 reduces by 75% the data the GPU must read per pass.
- **Parallelism (Model Parallelism):** **Tensor Parallelism** fragments the calculations of each layer across multiple GPUs in parallel; **Pipeline Parallelism** distributes layers sequentially (GPU 1 handles layers 1-40, GPU 2 layers 41-80).
- **Kernel Optimization (Flash Attention):** Reduces back-and-forth data trips between slow memory (HBM) and fast memory (SRAM) of the chip, dramatically accelerating attention calculation with long texts.

#### 2. Algorithmic Strategies (Optimize Logic)

- **KV Caching:** The most important optimization. Instead of recalculating the entire prompt from scratch for each new word, saves previous calculation results in a cache.
- **Speculative Decoding:** A small, fast "draft" model (1B) tries to guess the next 5-6 words; the large model checks them all at once.
- **Continuous Batching:** As soon as one request ends, a new one enters immediately without waiting for others. The GPU always works at full capacity.

---

## 0.6 Quantization

Quantization is the process of reducing the bits used to represent each model parameter — from 16-bit (FP16/BF16) to 8-bit (INT8), 4-bit (INT4) or even less.

**Why it works:** Large Language Models are incredibly resilient mathematical structures. Reducing numerical precision introduces statistical "noise," but the model tolerates these approximations without dramatic quality degradation.

**Why it's vital — The Bandwidth Wall:** In inference, the bottleneck is almost never GPU compute power, but the speed at which data travels from memory (VRAM) to compute cores.

> **Engineering Logic:** If the GPU must read 140 GB (FP16) versus 35 GB (INT4), the second operation will be theoretically **4× faster**.

**The Trade-off:**

| **Precision** | **Quality Impact** | **Ideal Use** |
|---|---|---|
| FP16 / BF16 | None (Original) | Critical sectors: Medicine, Legal, Scientific Research |
| INT8 | Minimal (1-3% drop) | Enterprise standard for balanced cost and fidelity |
| INT4 | Visible (5-10% drop) | Consumer chatbots, local assistants, large-scale deployment |

#### Technical Box: Memory-Bound vs Compute-Bound

LLM inference is almost always **Memory-Bound**, especially during decode. The processor is much faster than memory — the real limit is the bandwidth of the channel between VRAM and compute cores. Quantization is the most effective optimization: it doesn't speed up the physical channel, but reduces the volume of data to transport.

---

## 0.7 From Training to Serving

**Training:** Managed by the research team. Uses BF16 precision. A 70B model in training occupies 420 GB of memory (140 GB for weights, 280 GB for optimization structures). This phase produces the checkpoint.

**Release checkpoint:** The public distribution format. Weights saved in FP16 or BF16, typically in safetensors or GGUF. Immutable — this is what appears on HuggingFace.

**Inference optimization:** Where you take control. You download the checkpoint and convert it to the precision you need. This conversion is called post-training quantization.

The decision on which precision to use depends on your hardware constraints and business requirements:
- Only 40 GB VRAM? You must use INT4.
- Can afford 80 GB but want to maximize concurrent users? INT8 is a good compromise.
- Quality is critical and you have 160 GB? Use FP16.
- Want to balance everything? Use mixed formats: critical parameters in FP16, rest in INT4.

---

## 0.8 Why Internal Mechanics Matter

The difference between an efficient and an inefficient deployment is not (only) in choosing the most powerful GPU, but in understanding which resource saturates the workload.

**Concrete example:** A 70B in FP16 on H100 (80 GB VRAM) cannot load the weights (140 GB > 80 GB). Obvious solution: quantize to INT8 (70 GB). The model now fits in VRAM.

But after loading it, the GPU operates at 5% utilization during generation. Why?

It's not a compute power problem — the H100 has abundant compute. It's a bandwidth problem: the cores are ready to calculate but continuously waiting for parameters to arrive from VRAM. The GPU isn't underpowered, it's *waiting*. This is the Memory-Bound workload in action.

Buying an even more powerful GPU would not solve anything. The only effective lever is reducing the volume of data to transport: quantization, optimized formats, architectures that minimize memory reads.

# Chapter 1: How a Model Behaves

## 1.1 The Two Physics: Prefill vs Decode

**Prefill (Initialization):** The model reads the prompt (e.g., 1000 tokens) all at once. The GPU operates in **Compute-bound** mode: cores are saturated (60-70% utilization) because they have massive parallel data to work on.

**Decode (Generation):** The model generates one token at a time. We enter **Memory-bound** mode. Speed is dictated exclusively by how fast VRAM "fires" data to the cores.

---

## 1.2 The Memory Wall: Why the GPU Sits at 5%

Take the simplest concrete case: the **first generated token** of a new conversation, with **a single request in processing (batch=1)**. At this moment the KV cache is still empty.

Take a 70B model in INT8 (70 GB):

**1. Read Time:** An H100 has memory bandwidth of 3.35 TB/s (3350 GB/s). Reading 70 GB of weights requires:

```
70 GB ÷ 3350 GB/s = 0.0209 s ≈ 21 ms per token  [batch=1, empty KV cache]
```

**2. Compute Time:** H100 cores execute the math on that single token in less than 1 ms.

**3. The Result:**

```
1 ms work ÷ 22 ms total ≈ 4.5% utilization  [batch=1]
```

*4.5% is the most favorable case: first token, single request, empty cache. This is the theoretical maximum utilization ceiling in single-request decode mode.*

**Hardware Comparison: Bandwidth Determines Performance**

| Hardware | Bandwidth | Token/s (70B INT8, batch=1) |
|---|---|---|
| NVIDIA H100 | 3350 GB/s | 45–50 (Reference) |
| NVIDIA A100 | 2000 GB/s | 27–30 (~60% of H100) |
| RTX 4090 | 1008 GB/s | 13–15 (Consumer limit) |

*Indicative values at batch=1 — with multiple simultaneous requests the picture changes significantly.*

**Operational Insight:** If your load is 90% chat (heavy decode), buying the GPU with more TFLOPS is a mistake. You need to look at GB/s. A GPU with half the compute power but twice the memory bandwidth will generate responses twice as fast — at equal simultaneous requests.

---

## 1.3 Anatomy of a Live Forward Pass: Weights, KV Cache, and Batching

**The two loads on the memory channel:**

*Parameters (Fixed Weight):* 70 GB that traverse the channel every decode cycle. With 8 simultaneous requests (batch=8), these 70 GB are read once to serve all 8 — the fixed cost amortizes across the entire group.

*KV Cache (Dynamic Weight):* Did not exist at token 1. From token 2 onwards it grows at every step and is **individual per request** — not shared, not amortized.

**Cache explosion during generation (70B model):**

```
Cache per token = 2 × num_layers × hidden_dim × bytes_per_param
2 × 80 × 8192 × 2 bytes = 2.6 MB per token per request
```

**Forward pass comparison with varying context and concurrent requests:**

```
Token 1, batch=1 (empty cache):
  Weights:    70.0 GB
  KV cache:    ~0.0 GB
  Total:      70.0 GB ÷ 3350 GB/s ≈ 21 ms → 4.5% utilization

Token 4000, batch=1:
  Weights:    70.0 GB
  KV cache:  10.4 GB  (1 req × 2.6 MB × 4000 tokens)
  Total:      80.4 GB ÷ 3350 GB/s ≈ 24.0 ms → 4.0% utilization

Token 4000, batch=10:
  Weights:    70.0 GB  (read once for all 10 requests)
  KV cache: 104.0 GB  (10 reqs × 2.6 MB × 4000 tokens)
  Total:     174.0 GB ÷ 3350 GB/s ≈ 52 ms → 1.9% utilization
```

### Scenario: 50 users @ 4000 tokens each

- **Cache per user:** 10.4 GB
- **Total system cache:** 50 × 10.4 GB = 520 GB

With contiguous allocation, the KV cache of every user must coexist in VRAM simultaneously. The real constraint is not aggregate throughput but available VRAM.

---

## 1.4 Batching: Amortizing the Fixed Cost

**Definitions**

A **sequence** is the inference engine's unit of work: a single request in processing, with its prompt, already-generated tokens, and associated KV cache.

A **batch** is the set of sequences the engine processes simultaneously in a single forward pass. The **batch size** is the number of sequences in that group.

```
Request (application level)
    └── Sequence (engine level: prompt + generated tokens + KV cache)
            └── Batch (N sequences processed in a single forward pass)
```

**The math of batching (70B INT8 model):**

```
Weight cost per sequence = 70 GB ÷ N sequences in batch

batch=1:    70.0 GB per sequence
batch=8:     8.75 GB per sequence
batch=32:    2.19 GB per sequence
```

**The KV cache does not amortize.** It is individual by definition. Every sequence carries its own cache, proportional to its context length, and that cache must reside in VRAM for the duration of the sequence.

**Batch size tradeoffs:**

*Per-sequence latency:* Every forward pass processes N sequences simultaneously, which means each sequence waits for the entire batch to complete the step before receiving its token.

*VRAM saturation:* With N active sequences, their N caches must coexist in VRAM at the same instant.

*Diminishing returns:* The throughput gain per additional sequence is not linear.

```
Indicative marginal throughput gain:

batch=1  → batch=2:   +80-90%
batch=2  → batch=4:   +40-50%
batch=4  → batch=8:   +20-30%
batch=8  → batch=16:  +10-15%
batch=16 → batch=32:  +5-10%
```

**Batch size and sequence length: the combined constraint**

```
Sequences sustainably = 
    VRAM available for KV cache ÷ (2.6 MB × average sequence length)
```

High batch size requires short sequences, or more VRAM. This is the fundamental constraint every production configuration must balance.

# Chapter 2: What Makes a Model Expensive

## 2.1 The Model's Short-Term Memory

During generation, the model does not work in a vacuum: every produced token depends on all previous ones. To avoid recalculating the entire context at every new token, the system uses the **KV Cache**.

- **The Model:** transforms each token into two mathematical vectors (**K**ey and **V**alue) representing that word's contribution to the sequence context.
- **The Serving System:** stores these vectors in GPU VRAM.
- **The Inference Engineer:** must manage the space occupied by these vectors. Every added token is not just a mathematical operation — it is a byte cost that accumulates for the entire duration of the sequence.

---

## 2.2 The Economic Cost: How Much Does a Token Cost?

For a model like Llama 3 70B:

```
Bytes per token = 2 × num_layers × hidden_dim × bytes_per_param
2 × 80 × 8192 × 2 = 2,621,440 bytes ≈ 2.6 MB per token
```

A 4000-token conversation requires **10.4 GB** of cache. With 50 simultaneous sequences at that context level, the cache alone occupies **520 GB** — dwarfing the 70 GB of model weights.

#### Technical Box: The GQA Trick

Many modern models use **Grouped Query Attention**. Instead of saving cache for every attention head, keys are shared among groups — drastically reducing per-token cost.

Llama 3 70B uses 8 KV heads on 64 attention heads: the reduction factor is exactly 8×, bringing per-token cost from ~2.6 MB to ~0.32 MB.

---

## 2.3 Fragmentation and PagedAttention

Before PagedAttention, cache was managed as contiguous blocks, creating two problems:

**Internal fragmentation:** allocate space for 4000 tokens, sequence uses only 200 — 95% of memory occupied but empty.

**External fragmentation:** 20 GB free but distributed in scattered blocks — cannot admit a new sequence needing 10 GB contiguous, even if total space exists.

**PagedAttention** resolves both by applying OS virtual memory concepts:

1. Cache is split into small fixed pages (e.g., 16 tokens per page)
2. Physical pages can be scattered anywhere in VRAM — they don't need to be contiguous
3. A **Page Table** per sequence tracks where its conversation fragments are

**Result:** VRAM utilization goes from ~30% to ~90%. The same GPU can serve 2-3× more simultaneous sequences.

---

## 2.4 Prefix Sharing: Sharing the Past

With contiguous allocation, memory is physically separate per sequence. With pages, pointing different sequences' Page Tables to the same physical pages for the common prefix is sufficient.

**Typical use case:** a chatbot with a fixed 500-token system prompt shared by 100 simultaneous sequences.

Without Prefix Sharing: those 500 tokens are computed and stored 100 times.

With Prefix Sharing: computed and stored **once**. All 100 sequences read the same physical pages until they start generating their individual responses — from that point their caches diverge and become independent.

---

## 2.5 Cache Quantization (FP8 / INT8)

The third lever is reducing the weight of each cached token by going from FP16 to INT8 or FP8:

| Characteristic | Weight Quantization (INT4) | KV Cache Quantization (FP8/INT8) |
|---|---|---|
| Memory Saving | 4× (Static) | 2× (Dynamic) |
| Degradation | 5-10% (Permanent) | <1% (Temporary on session) |
| Bandwidth Benefit | Speeds up weight loading | Reduces data read in decode |

**Why is degradation so low?** KV cache values are intermediate calculation results — not the deep model weights. Small precision loss on these temporary values has negligible impact on final response quality.

**Practical rule:** if you are short on VRAM, quantizing the cache is the first move. It does not touch model intelligence — only reduces precision of temporary data that exists only for the duration of the sequence.

---

## 2.6 The Trade-off: Capacity vs Bandwidth

**Capacity (how many sequences fit):** solved by PagedAttention and Prefix Sharing — less space occupied per sequence, more sequences in the same VRAM.

**Bandwidth efficiency (how fast you respond):** solved by quantization — less data to read per forward pass, more tokens per second.

**Example of conflict — Prefix Sharing:**

On the capacity front: 100 sequences share the same system prompt. Instead of 100 cache copies, only one exists — 99% saving for that block.

On the bandwidth front: no saving. To generate each sequence's response, the GPU must still read those data from VRAM per forward pass.

**In summary:** PagedAttention and Prefix Sharing let you fit more sequences in the GPU. Quantization lets you respond faster to those sequences. They are complementary — in production they are used together.

---

#### Box: Advanced KV Cache Management Techniques

**KV Cache Offloading (CPU/NVMe):** When VRAM saturates, cache of less active sequences is moved to system RAM or NVMe storage, reloaded when that sequence becomes active again. The trade-off is latency: CPU→GPU bandwidth (typically 32-64 GB/s on PCIe) is two orders of magnitude lower than internal VRAM bandwidth. Useful for sequences with very long contexts and low access frequency. Directly measured in Dielabs benchmarks with LMCache on Qwen3-8B-AWQ — results show concrete benefits under real memory pressure.

**Sliding Window Attention:** Instead of maintaining cache for all context tokens, the model maintains only a window of the N most recent tokens. Used in Mistral 7B v0.1.

**StreamingLLM:** Variant of sliding window that also maintains the very first context tokens ("attention sinks") alongside the recent sliding window. Achieves stability with arbitrarily long contexts at near-constant VRAM cost.

**Token Eviction / Selective Importance:** Instead of discarding tokens by position, discards those that received less attention in past iterations. Techniques like H2O (Heavy Hitter Oracle) dynamically select which tokens to keep.

# Chapter 3: Where the Model Lives

The layer separation is a diagnostic protocol. The value is not in knowing which layer a component belongs to, but in knowing **which layer has the authority to modify the behavior** generating the symptom.

A problem is solved at the correct layer when the fix requires no modifications to adjacent layers and creates no regressions in layers above.

## L0 — The Silicon

The hardware choice is not just about TFLOPS, but about balancing **Capacity** (how much VRAM) and **Bandwidth** (how fast the pipe).

- **NVIDIA H100:** 80 GB VRAM, 3.35 TB/s Bandwidth. The Ferrari: generates tokens fast but limited capacity for giant models.
- **DGX Spark / Unified Systems:** 128-256 GB VRAM, 273 GB/s Bandwidth. The truck: can load enormous models (high capacity) but generates tokens much more slowly.

**Interconnects:** Transform isolated GPUs into a coherent system.

- **NVLink (900 GB/s):** The private "tunnel" **intra-node**. The only one fast enough for **Tensor Parallelism** thanks to ultra-low latency (~1-2 μs). ~15-20× faster than PCIe.
- **InfiniBand (~200 GB/s):** Dedicated network fabric, separate from Ethernet. Very low latency (~1 μs), high performance, but requires costly proprietary infrastructure. Used in high-end HPC and AI clusters.
- **RoCE — RDMA over Converged Ethernet (25-100 GB/s):** RDMA implemented on standard Ethernet. Same mechanism as InfiniBand — direct memory-to-memory transfer between servers bypassing the CPU — but on existing Ethernet infrastructure. Slightly higher latency than InfiniBand, significantly lower cost. The prevalent choice for **inter-node** linking in modern AI datacenters.
- **PCIe (32-64 GB/s):** The system bus — connects GPU to CPU and system RAM. Present in any x86 server. Too slow for coordinating distributed calculations between GPUs.

## L1 — Drivers

The Layer 1 is the software layer between the inference framework and the silicon.

**Driver (NVIDIA):** The foundation of the entire stack. The installed driver version defines which CUDA version is supportable. A container requiring CUDA 12.x will not start on a host with driver 515.x — minimum required version is 525.x.

**CUDA Runtime & API:** Translates framework operations (PyTorch, vLLM) into GPU-executable instructions.

**NCCL** (NVIDIA Collective Communications Library): Manages synchronization and data transfer between multiple GPUs — both intra-node via NVLink and inter-node via InfiniBand/RoCE.

## L2 — Runtime & Packaging

L2 isolates and distributes the execution environment — the world of containers.

Components: containerd or Docker as container runtime, nvidia-container-runtime as bridge between container and host GPU driver, CUDA-based images as base for all environments, compiled libraries (PyTorch, TensorRT-LLM, vLLM) distributed as part of the image.

L2 decisions are binding for all higher layers. This layer is also the source of one of the most frequent problems in multi-node environments: **image drift**. Model works on node A and crashes on node B for no apparent reason. The cause is almost always a different image, library, or runtime version between nodes.

## L3 — Inference Backend (Execution Engine)

L3 is the computational heart. This is where the model actually executes the forward pass.

Examples: vLLM, TensorRT-LLM, llama.cpp, TGI. Responsibilities: loading weights to VRAM, KV cache management, GPU memory allocation, CUDA kernel execution, continuous batching if integrated in the backend.

**Important detail:** this layer does not know about users. It works on tensors and batches. Abstraction toward the outside world happens in higher layers.

## L4 — Serving & Orchestration

L3 decides *how* to compute. L4 decides *where* and *when* to run it.

Components: Kubernetes, Slurm, Nomad. Functions: GPU-specific scheduling, autoscaling in response to load, pod lifecycle management, health checks, resource quota management.

The critical distinction: a perfectly optimized L3 backend can appear unusable if L4 schedules it poorly. The typical problem is GPU idle with a full queue — not a backend problem, a scheduler policy problem.

## L5 — API Gateway / Edge

L5 is the layer that protects the backend from everything external.

Functions: rate limiting, authentication, request validation, context cap enforcement, multi-tenant isolation.

Without explicit limits, a single tenant can saturate backend VRAM by sending excessively long prompts. Context cap enforcement is not a luxury feature — it is the primary defense against externally-induced OOM.

## L6 — Client

L6 is the origin of real load.

**Operational fact:** a **stateless client multiplies prefill cost**. If every request sends the entire conversation history instead of maintaining state, the backend must recalculate prefill from scratch every time. The result is high TTFT that appears to be a GPU problem but is an application design problem.

# Chapter 4: How to Optimize the Model

## 4.1 Batching: The Efficiency of Shared Journey

**The math of batching (70B INT8 model):**

- **Single request:** Read 70 GB → generate 1 token → time: **21 ms**
- **Batch of 4 requests:** Read 70 GB **once** → generate 4 tokens → time: **21 ms**
- **Result:** Throughput goes from 1 to 4 tokens/21 ms. A **4× speedup** at equal bandwidth.

**Evolution: From Static to Continuous**

1. **Static Batching:** The system waits for 8 people before starting the bus. If one user writes a novel and others a word, seats remain occupied and unused until the last one finishes.
2. **Continuous Batching:** As soon as a user finishes, they get off the bus. At the next stop, a new user from the queue immediately boards. The GPU always travels full.

**Chunked Prefill: The "Hiccup" Problem**

A long prompt (Prefill) "clogs" the system, creating jitter for users already in generation (Decode).

An atomic 1000-token prefill blocks the system for ~50 ms (3 lost decode cycles). With **Chunking**, it is split into 4 pieces of 250 tokens (~12 ms each), interleaved with other users' generation. Result: zero "hiccups".

> **Layer Responsibility:** L3 (Inference Backend)
> **Config Parameter** (vLLM): `max_num_seqs`, `enable_chunked_prefill=True`

---

## 4.2 Parallelism: Splitting the Model Across GPUs

**Intra-Node: Tensor Parallelism (TP)**

Split the **matrices of each layer** across GPUs in the same server.

- Requires continuous synchronization via **NVLink** (900 GB/s)
- Overhead: 80 layers × 2 μs latency = 160 μs (~1.6% of total time). Nearly invisible.

**Inter-Node: Pipeline Parallelism (PP)**

Split the model **sequentially** across different servers (e.g., GPU 1: layers 1-40; GPU 2: layers 41-80).

- Uses **InfiniBand/RoCE** inter-node fabric
- **RDMA:** Allows GPU on Node A to write data directly into VRAM of GPU on Node B, bypassing CPU and OS. Without RDMA overhead is 20 μs; with RDMA it drops to **1-5 μs**.

> **Layer Responsibility:** L0 (NVLink/InfiniBand fabric) + L3 (Inference Backend)
> **Config Parameter** (vLLM): `tensor_parallel_size=N`, `pipeline_parallel_size=N`

---

## 4.3 Prefix Caching: Don't Repeat Yourself

In RAG or Enterprise Chatbot systems, many requests share the same System Prompt.

- **Without Caching:** Every request recalculates common token prefill.
- **With Caching:** The system saves the prefix KV Cache in VRAM. If a request with identical prefix arrives, it loads instantly.
- **Impact:** **Time to First Token (TTFT)** drops dramatically (up to 80%).

> **Layer Responsibility:** L3 (Inference Backend)
> **Config Parameter** (vLLM): `enable_prefix_caching=True`

---

## 4.4 Speculative Decoding: The Guesser and the Master

Decode is sequential. This technique breaks the limit using a "Guesser."

1. **Draft Model (The Guesser):** Small (1B-7B), very fast, imprecise. Bets on the next 5 words.
2. **Target Model (The Master):** Large (70B), slow, infallible. Checks all 5 words at once.

If the draft model guesses correctly (very likely for articles and conjunctions), multiple words are generated in the time of one.

> **Layer Responsibility:** L3 (Inference Backend)
> **Config Parameter** (vLLM): `speculative_model="draft_model_name"`, `num_speculative_tokens=5`

---

## 4.5 Decision Cheat Sheet

| **If the problem is...** | **Technique** | **Layer** | **Config Parameter** |
|---|---|---|---|
| Full queue, underutilized GPU | Continuous Batching | L3 | `max_num_seqs` ↑ |
| Model too large for one GPU | Tensor Parallelism | L0 + L3 | `tensor_parallel_size=N` |
| High TTFT (repeated prefixes) | Prefix Caching | L3 | `enable_prefix_caching=True` |
| Low TPS, slow decode | Speculative Decoding | L3 | `speculative_model` config |
| Multi-node distributed model | Pipeline Parallelism | L0 + L3 | `pipeline_parallel_size=N` |

# Chapter 5: Architecture and Economics

## 5.1 Deployment Topologies: Choosing Your Constraint

In inference, you cannot have everything. You must choose which "bottleneck" to accept in exchange for other advantages.

### A. Datacenter Scale-Out (Throughput Ferrari)

- **Architecture:** Multi-node cluster (8× GPU) connected via NVLink internally and InfiniBand externally.
- **Strength:** Generation speed (TPS) and massive scalability.
- **Constraint:** Extreme complexity. Requires a dedicated SRE team.

### B. Unified Memory Workstation (Capacity Truck)

- **Architecture:** CPU and GPU share the same RAM.
- **Examples:** Apple M3 Max (128 GB unified, ~400 GB/s), NVIDIA Grace Hopper (512 GB unified, 450 GB/s), NVIDIA DGX Spark GB10 (128 GB unified, 273 GB/s).
- **Strength:** Enormous capacity (175B+ models) at a fraction of the cost.
- **Constraint:** **Bandwidth**. Unified memory is 7-8× slower than HBM. If an H100 generates 50 tokens/s, here you will get 6-7.

### C. Cloud API (Public Bus)

- **Architecture:** Managed services (OpenAI, AWS Bedrock). Pay per use.
- **Constraint: Noisy Neighbor Effect.** In congestion, TTFT can go from 300 ms to 1200 ms (4× degradation).

---

## 5.2 Network Topology: The Cost of Distance

- **NVLink (L0 - Internal):** 900 GB/s. Tensor Parallelism is "free" (1.5% overhead). GPUs act as a single brain.
- **InfiniBand (L0 - External):** 400 Gb/s spec (NDR), effective bidirectional throughput **25-50 GB/s**. TP becomes costly. Use **Hybrid** approach: TP inside the server, **PP** between servers via RDMA.
- **Ethernet (L0 - Standard):** 100 GbE (~12.5 GB/s). 10-50 μs latency. TP unusable (>25% overhead). Only PP with micro-batching is sustainable.

---

## 5.3 System Metrics: Beyond Average Speed

Do not be fooled by the mean (p50). In production, a system is only as good as its **p99**.

1. **TTFT (Responsiveness):** Time to first token. **Target: < 500 ms (p95).**
2. **TPOT / ITL (Fluidity):** Time between tokens. **Target: < 50 ms (p95).** Unstable ITL causes "stuttering text" effect.
3. **Goodput:** The truth metric. The number of generated tokens that respect SLOs.

> **Efficiency = Goodput / Throughput.** Increasing batch size increases throughput, but if it destroys latency, Goodput drops.

## 5.4 Cost Modeling: The Token Economy

Analysis on an 8× H100 server:

- **CAPEX:** $200,000 amortized over 3 years → **$182/day**
- **OPEX (Power & Cooling):** 5 kW IT load with PUE 1.4 → 7 kW total, at $0.12/kWh → **$20/day**
- **Total Daily Cost:** **$202/day**

**Impact of Inference Engineering:**

## 5.5 Decision Tree: Which Architecture for You?

**START: Identify your primary constraint**

1. **Volume < 10M tokens/month** → **Cloud API.** Zero CAPEX, immediate time-to-market.
2. **Volume 10-100M tokens/month:**
   - Latency critical + technical team → **Single-node self-hosted** (8× H100)
   - No SRE team → **Hybrid** (Base self-hosted + Cloud burst)
3. **Volume > 100M tokens/month:**
   - Privacy/Compliance critical → **Private Datacenter** (Multi-node InfiniBand)
   - Maximum efficiency + SRE team → **Large-scale cluster** (TP + PP)

## Fundamental Principles

### 1. In decode, bandwidth always beats compute

Don't buy GPUs for TFLOPS. Buy for GB/s memory bandwidth and total GB of VRAM. Memory bandwidth determines tokens/second in decode, which is >90% of total inference time.

### 2. The KV cache is the hidden cost

When sizing infrastructure, do not calculate only parameter size. Calculate:

```
Total memory = Parameters + (Simultaneous users × Average context × Cache per token)

Cache_per_token depends on architecture:
- Dense model 70B: 2.6 MB/token
- GQA model 70B: ~0.32 MB/token (8× reduction)
```

### 3. Batching is the only universal optimization

Continuous batching always works. It improves throughput by amortizing memory access across multiple users without degrading individual latency. If you implement one optimization, make it this.

### 4. Measure percentiles, not means

A system with p50 latency 200 ms but p99 latency 5000 ms is a failed production system. Users live in the long tail of the distribution.

### 5. Cost per token is the only business metric

```
Unoptimized deployment:
- Throughput: 300 token/sec goodput
- Cost: $7.80 per 1M token

Optimized deployment (vLLM + batching + quantization + caching):
- Throughput: 1000 token/sec goodput
- Cost: $2.34 per 1M token

Same hardware, 70% saving per token, 3.3× more users served.
```

### 6. Complexity lives somewhere, it never disappears

No "simple and performant and economical" simultaneously. Choose which two you get:

- **Simple + Performant:** cloud managed services, pay premium cost
- **Performant + Economical:** datacenter self-hosted, pay complexity ops
- **Simple + Economical:** workstation scale-up, pay throughput limitation

## The Diagnostic Loop

When facing a production problem, apply this systematic workflow:

**1. Identify the bottleneck:** which resource saturates?
```
Check metrics:
- GPU utilization (target: 60-90%)
- VRAM utilization (target: 80-90%)  
- Memory bandwidth (target: 80-95%)
- Queue depth (target: controlled, no monotonic growth)
```

**2. Map to the stack layer:**
```
- GPU idle + full queue → L4 (Scheduler)
- Immediate OOM → L3 (Backend memory)
- Progressive OOM → L5 (Gateway limits)
- High latency + active GPU → L0 (Hardware bandwidth)
- p99 jitter spikes → L4 (Batching policy)
```

**3. Choose the appropriate technique:**
```
- Full queue → Continuous batching (max_num_seqs ↑)
- Model doesn't fit → Tensor Parallelism or Quantization
- High TTFT with repeated prefixes → Prefix caching
- Low throughput, latency OK → Speculative decoding
- Multi-server latency → Pipeline Parallelism
```

**4. Measure impact** (before-after comparison on p50, p95, p99 latency; throughput; cost per token).

**5. Iterate** — the bottleneck has shifted. Which resource limits now?

## Emerging Directions (Watch This Space)

**Disaggregated Serving:** Physical separation between prefill cluster (compute-intensive) and decode cluster (memory-intensive), with KV cache transferred via RDMA. Justified only at massive scale.

**Sub-4-bit Quantization with Mixed Precision:** INT3, INT2, even binary weights for non-critical layers. Research active, some production deployment with INT3 on specific models.

**End-to-End Kernel Fusion:** Compilers (Triton, XLA, TVM) fuse multiple operations into a single kernel launch. In production in framework-specific deployments (TensorRT-LLM aggressive fusion, vLLM via Triton kernels).

## The Craft of Inference Engineering

The AI inference engineer is a unique hybrid figure:

- Not an ML researcher (receives frozen checkpoints from training team)
- Not a traditional software engineer (does not write business logic)
- Not a generic SRE (manages workloads with physics completely different from stateless web services)

A **specialized systems architect** for workloads with unique characteristics:

- **Memory-bound** instead of compute-bound (95% time in memory access)
- **Stateful with linear growth** (KV cache accumulates during conversation)
- **Latency-sensitive with high throughput requirements** (p95 TTFT <500 ms but serves 1000+ users)
- **Cost dominated by hardware CAPEX** not labor (GPU depreciation, not eng salaries)
- **Utilization target 80-90%** not 30-40% (hardware too expensive to idle)

The AI infrastructure that matters is not only better models (research labs produce breakthroughs), but existing models served better (infrastructure engineers extract maximum value). Researchers push the frontier of what is possible; infrastructure engineers make it economically sustainable at scale.
